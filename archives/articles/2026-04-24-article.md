# 大模型Function Calling实战

## 核心结论

**大模型Function Calling的本质是将LLM的语义理解能力与外部系统的结构化交互能力进行桥接，通过JSON Schema定义工具接口，让模型自主决定何时调用、传入什么参数，从而实现从“对话机器”到“任务执行器”的跃迁。**

---

## 为什么重要

在AI工程师的成长路径中，掌握基础Prompt工程后，最常见的瓶颈是：**模型只能“说”，不能“做”**。你问它“帮我查一下今天北京的天气”，它只能回复一段文字，而无法真正调用天气API。Function Calling正是打破这一瓶颈的关键技术。

从技术演进角度看，Function Calling位于大模型应用开发的**核心枢纽位置**。它向上承接Prompt工程中的指令理解、意图识别能力，向下连接API、数据库、工具链等外部系统。没有Function Calling，LLM只能是一个“高级聊天机器人”；有了它，LLM才能成为真正的“智能体”——能够理解用户意图、自主选择工具、执行复杂任务。

对于AI工程师而言，Function Calling是构建Agent系统、RAG流水线、自动化工作流的基础设施。你可能会发现，80%的企业级LLM应用，核心都是在“模型理解自然语言→映射到结构化函数调用”这个循环上做文章。掌握它，意味着你从“调模型API”进化到了“设计模型驱动的系统架构”。

---

## 核心内容

### 一、工具定义：给模型一把“瑞士军刀”

#### 概念说明

工具定义就是**告诉模型“你能做什么”**。你需要把外部函数（如天气查询、数据库查询、发送邮件）描述给模型，让模型知道：存在哪些能力、每个能力需要什么参数、参数是什么类型。

#### 原理解析

大模型本身不具备调用函数的能力。Function Calling的工作流程是：

1. 开发者定义一组函数描述（JSON Schema格式），随用户消息一起发给模型
2. 模型分析用户意图，判断是否需要调用某个函数
3. 如果需要，模型返回一个**结构化对象**（函数名 + 参数），而不是自然语言回复
4. 开发者解析这个对象，执行实际函数调用，将结果返回给模型
5. 模型基于函数执行结果，生成最终回复

关键洞察：**模型不执行函数，它只决定“该调用哪个函数”和“参数应该是什么”。**

#### 代码示例

```python
import json
from openai import OpenAI

client = OpenAI()

# 定义工具（函数描述）
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取指定城市的当前天气信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "城市名称，如'北京'、'上海'"
                    },
                    "unit": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"],
                        "description": "温度单位，默认为celsius"
                    }
                },
                "required": ["city"]
            }
        }
    }
]

# 用户消息
messages = [{"role": "user", "content": "今天北京天气怎么样？"}]

# 调用模型，传入工具定义
response = client.chat.completions.create(
    model="gpt-4",
    messages=messages,
    tools=tools,
    tool_choice="auto"  # 让模型自动决定是否调用
)

# 解析模型返回的调用请求
if response.choices[0].message.tool_calls:
    tool_call = response.choices[0].message.tool_calls[0]
    function_name = tool_call.function.name
    arguments = json.loads(tool_call.function.arguments)
    print(f"模型决定调用: {function_name}")
    print(f"参数: {arguments}")
```

### 二、JSON Schema：定义“完美契约”

#### 概念说明

JSON Schema是描述函数参数的**标准化语言**。它不是Python函数签名，而是一个JSON对象，告诉模型：参数有哪些、每个参数的类型、取值范围、是否必须。**这是模型与外部系统之间的契约。**

#### 原理解析

为什么必须用JSON Schema？因为大模型是**概率生成器**，它输出的是token序列。如果不提供严格的结构化约束，模型可能会生成“city=北京”或“city=’北京’”等各种格式，导致解析失败。JSON Schema通过以下机制确保可靠性：

- **类型约束**：`type: "integer"` 确保模型不会传入字符串
- **枚举约束**：`enum: ["celsius", "fahrenheit"]` 限制取值范围
- **必填约束**：`required: ["city"]` 确保关键参数不缺失
- **嵌套结构**：支持对象嵌套，描述复杂参数

**最佳实践**：参数描述要详细、具体。模型通过description字段理解参数含义，描述越清晰，模型生成准确参数的概率越高。

#### 代码示例

```python
# 复杂工具定义：多参数、嵌套结构、枚举约束
complex_tools = [
    {
        "type": "function",
        "function": {
            "name": "book_flight",
            "description": "预订航班，支持国内单程和往返航班",
            "parameters": {
                "type": "object",
                "properties": {
                    "origin": {
                        "type": "string",
                        "description": "出发城市三字码，如'PEK'、'SHA'"
                    },
                    "destination": {
                        "type": "string",
                        "description": "到达城市三字码"
                    },
                    "travel_type": {
                        "type": "string",
                        "enum": ["one_way", "round_trip"],
                        "description": "行程类型：单程或往返"
                    },
                    "passengers": {
                        "type": "array",
                        "items": {
                            "type": "object",
                            "properties": {
                                "name": {"type": "string"},
                                "age": {"type": "integer", "minimum": 0, "maximum": 120},
                                "id_type": {
                                    "type": "string",
                                    "enum": ["id_card", "passport"]
                                }
                            },
                            "required": ["name", "age", "id_type"]
                        },
                        "description": "乘客信息列表"
                    }
                },
                "required": ["origin", "destination", "travel_type", "passengers"]
            }
        }
    }
]

# 验证JSON Schema的约束效果
test_arguments = {
    "origin": "PEK",
    "destination": "SHA",
    "travel_type": "one_way",
    "passengers": [
        {"name": "张三", "age": 30, "id_type": "id_card"}
    ]
}
```

### 三、错误处理：让调用“优雅地失败”

#### 概念说明

Function Calling不是“一次成功”的魔法。模型可能生成错误的参数名、缺失必填字段、或者误解用户意图。**错误处理不是事后补救，而是系统设计的一部分。**

#### 原理解析

常见的错误场景及应对策略：

| 错误类型 | 表现 | 处理策略 |
|---------|------|---------|
| 参数格式错误 | 参数名拼写错误、类型不匹配 | 捕获异常后，将错误信息作为系统消息返回给模型，让模型修正 |
| 必填参数缺失 | 缺少required字段 | 检查参数完整性，缺失时要求模型补充 |
| 意图误判 | 用户没说调用函数，模型却调用了 | 设置`tool_choice: "none"`或通过Prompt约束 |
| 业务逻辑错误 | 参数值合法但不合理（如过去日期） | 在函数执行层验证，返回业务错误信息 |

**核心原则**：**永不信任模型的输出**。模型生成的结构化数据必须经过验证才能执行。

#### 代码示例

```python
import json
import jsonschema
from jsonschema import validate, ValidationError

def safe_execute_function(tool_call, function_registry):
    """
    安全的函数执行器：验证→执行→错误恢复
    """
    try:
        # 1. 解析模型输出
        function_name = tool_call.function.name
        arguments = json.loads(tool_call.function.arguments)
        
        # 2. 查找函数定义
        if function_name not in function_registry:
            raise ValueError(f"未知函数: {function_name}")
        
        func_def, func_impl = function_registry[function_name]
        
        # 3. 验证参数合法性
        try:
            validate(instance=arguments, schema=func_def["parameters"])
        except ValidationError as e:
            # 返回结构化错误信息给模型
            error_msg = f"参数验证失败: {e.message}"
            return {
                "status": "error",
                "error_type": "validation_error",
                "message": error_msg,
                "expected_schema": func_def["parameters"]
            }
        
        # 4. 执行业务函数
        result = func_impl(**arguments)
        return {"status": "success", "result": result}
        
    except json.JSONDecodeError:
        return {
            "status": "error",
            "error_type": "parse_error",
            "message": "无法解析模型返回的JSON参数"
        }
    except Exception as e:
        return {
            "status": "error",
            "error_type": "execution_error",
            "message": str(e)
        }

# 使用示例
def get_weather_impl(city: str, unit: str = "celsius"):
    # 实际API调用
    return {"temperature": 25, "condition": "晴"}

function_registry = {
    "get_weather": (tools[0]["function"], get_weather_impl)
}

# 模拟模型返回的错误参数
bad_tool_call = type('ToolCall', (), {
    'function': type('Function', (), {
        'name': 'get_weather',
        'arguments': '{"city": "北京", "unit": "invalid_unit"}'
    })
})()

result = safe_execute_function(bad_tool_call, function_registry)
print(result)
# 输出: {'status': 'error', 'error_type': 'validation_error', ...}
```

## 实践建议

1. **从小处着手**：先定义一个最简单的工具（如获取当前时间），跑通“定义→调用→执行→返回”的完整流程，再逐步增加复杂度

2. **工具描述要“贪婪”**：在description字段中尽可能详细地描述参数含义、格式、取值范围。**模型是通过description理解参数的**，而非通过参数名

3. **建立验证流水线**：在项目初期就引入jsonschema库进行参数验证，不要等到生产环境出问题再补救。每个函数调用都必须经过“格式验证→业务验证”两层检查

4. **错误信息要可消费**：当函数执行失败时，返回给模型的错误信息应该包含：错误原因、期望格式、可操作建议。让模型能够基于错误信息自行修正

5. **使用结构化日志**：记录每次Function Calling的完整链路：用户输入→模型决策→参数生成→验证结果→执行结果。这在调试和优化时至关重要
