# Multimodal RAG：多模态检索增强

## 核心结论

**多模态检索增强生成（Multimodal RAG）通过将图像、表格、图表等非文本数据纳入检索与生成流程，使大模型能够基于多模态信息进行推理和回答，是解决纯文本RAG在视觉信息、结构化数据理解上瓶颈的关键技术路径。**

## 为什么重要

在当前的AI工程实践中，RAG（检索增强生成）已经成为解决大模型知识更新、幻觉控制和领域问答的主流范式。然而，绝大多数RAG系统仍然局限于文本模态——它们将文档切片、向量化、检索，最终只把文本片段送入大模型。这种做法在面对包含图表、截图、扫描件、表格等丰富视觉信息的文档时，会遭遇严重的性能瓶颈：**图像中的趋势、对比关系、布局信息在文本抽取过程中大量丢失**，表格被扁平化为无序文本后丧失结构性，产品说明书中的示意图无法被有效利用。

对于AI工程师而言，掌握Multimodal RAG意味着能够构建真正理解“文档内容”的系统，而不是仅仅处理“文档中的文字”。这在金融研报分析、医疗影像报告、法律文档审阅、产品手册问答等场景中具有直接的价值。它要求工程师突破纯文本的思维定式，理解多模态嵌入、跨模态检索、视觉语言模型等核心组件，并将它们有机地整合到RAG架构中。

从技术成长路径看，Multimodal RAG是传统RAG系统与多模态大模型（如GPT-4V、Gemini Pro Vision）之间的桥梁。它既不需要依赖昂贵的端到端多模态大模型进行全量推理，又能显著提升系统对视觉信息的理解能力，是当前阶段投入产出比最高的技术方向之一。

## 核心内容

### 图像检索：从视觉特征到语义匹配

#### 概念说明

图像检索是Multimodal RAG的基础能力，它解决的核心问题是：**给定一张查询图像或一段文本描述，如何从大规模图像库中找到语义上最相关的结果**。这与传统RAG中的文本向量检索类似，但操作对象从文本变成了图像。

#### 原理解析

图像检索的核心流程分为三个步骤：

1. **图像编码**：使用视觉编码器（如CLIP的ViT、SigLIP、DINOv2）将图像转换为固定维度的特征向量。
2. **索引构建**：将所有图像的特征向量存入向量数据库（如FAISS、Milvus、Chroma）。
3. **相似度搜索**：对查询图像或文本进行编码，在向量空间中执行近似最近邻搜索。

关键点在于**对齐**：我们需要一个能够同时理解图像和文本的联合嵌入空间。CLIP模型通过对比学习实现了这一点——它将配对的图像-文本对拉近，将非配对的推远，从而创造出跨模态的语义空间。

```python
import torch
from PIL import Image
from transformers import CLIPProcessor, CLIPModel

# 加载CLIP模型
model = CLIPModel.from_pretrained("openai/clip-vit-base-patch32")
processor = CLIPProcessor.from_pretrained("openai/clip-vit-base-patch32")

def encode_image(image_path: str) -> torch.Tensor:
    """将图像编码为嵌入向量"""
    image = Image.open(image_path).convert("RGB")
    inputs = processor(images=image, return_tensors="pt")
    with torch.no_grad():
        embedding = model.get_image_features(**inputs)
    # L2归一化，便于余弦相似度计算
    return embedding / embedding.norm(dim=-1, keepdim=True)

def encode_text(text: str) -> torch.Tensor:
    """将文本编码为嵌入向量"""
    inputs = processor(text=text, return_tensors="pt", padding=True, truncation=True)
    with torch.no_grad():
        embedding = model.get_text_features(**inputs)
    return embedding / embedding.norm(dim=-1, keepdim=True)

# 示例：构建简单的图像检索
image_paths = ["chart1.png", "chart2.png", "table1.png"]
image_embeddings = torch.cat([encode_image(p) for p in image_paths])

query = "2024年Q2的营收趋势图"
query_embedding = encode_text(query)

# 计算相似度
scores = (query_embedding @ image_embeddings.T).squeeze(0)
best_idx = scores.argmax().item()
print(f"最匹配的图像：{image_paths[best_idx]}, 相似度：{scores[best_idx]:.4f}")
```

### 表格理解：结构化信息的无损检索

#### 概念说明

表格是文档中信息密度最高的元素之一，但也是纯文本RAG处理最薄弱的环节。表格理解的目标是在不丢失行列关系和数值语义的前提下，将表格转换为可检索、可推理的表示形式。

#### 原理解析

处理表格有三种主流策略，它们在信息保真度和实现复杂度上各有取舍：

**策略一：文本序列化**。将表格转换为Markdown格式或CSV文本，然后使用常规的文本嵌入模型进行检索。这种方式最简单，但会丢失表格的结构信息，且对于大型表格，序列化后的文本长度可能超出模型的上下文窗口。

**策略二：图像化处理**。将表格截图作为图像，使用视觉编码器进行检索。这种方式保留了表格的视觉布局，但无法直接获取精确的数值信息。

**策略三：结构化嵌入**。使用专门设计的表格编码器（如TAPAS、Table Transformer），将表格的行、列、单元格分别编码为结构化的向量表示，支持对特定行列或单元格的精确检索。

在实践中，推荐**混合方案**：对于小表格（<50行），使用Markdown序列化；对于大表格或复杂布局的表格，使用图像化处理，并在生成阶段结合OCR或表格解析工具提取关键数据。

```python
import pandas as pd
from typing import Dict, Any

def table_to_markdown(df: pd.DataFrame) -> str:
    """将DataFrame转换为Markdown格式的文本"""
    return df.to_markdown(index=False)

def table_to_structured_text(df: pd.DataFrame) -> str:
    """将表格转换为结构化描述文本"""
    lines = []
    # 表头信息
    headers = list(df.columns)
    lines.append(f"表格包含 {len(df)} 行，列名：{', '.join(headers)}")
    
    # 数值列统计
    for col in df.select_dtypes(include=['number']).columns:
        lines.append(f"列'{col}'：最小值 {df[col].min()}，最大值 {df[col].max()}，均值 {df[col].mean():.2f}")
    
    # 关键行摘要（前3行和后3行）
    lines.append("前3行数据：")
    lines.append(df.head(3).to_markdown(index=False))
    if len(df) > 6:
        lines.append("后3行数据：")
        lines.append(df.tail(3).to_markdown(index=False))
    
    return "\n".join(lines)

# 示例：表格检索的混合策略
def retrieve_table_context(table: pd.DataFrame, query: str) -> Dict[str, Any]:
    """根据查询返回表格的检索上下文"""
    # 策略1：Markdown序列化
    md_text = table_to_markdown(table)
    
    # 策略2：结构化摘要
    summary_text = table_to_structured_text(table)
    
    # 根据查询类型决定返回哪种表示
    if any(word in query for word in ["趋势", "对比", "分布", "占比"]):
        # 视觉分析类问题，返回结构化摘要
        return {"type": "summary", "content": summary_text}
    else:
        # 精确查询类问题，返回完整表格
        return {"type": "full_table", "content": md_text}
```

### 多模态嵌入：构建统一的检索空间

#### 概念说明

多模态嵌入是Multimodal RAG的核心技术，它解决的问题是：**如何让图像、文本、表格等不同模态的数据在同一个向量空间中具有可比性**。只有建立了这样的统一空间，我们才能实现跨模态的检索——例如，用户输入一段文字，系统能够返回最相关的图像。

#### 原理解析

多模态嵌入模型通常采用**双塔架构**：一个编码器处理图像（视觉塔），另一个处理文本（文本塔），通过对比学习将配对样本的嵌入向量拉近。CLIP是这一范式的代表，但近年来出现了更强大的替代方案：

- **SigLIP**：Google提出的改进版CLIP，使用Sigmoid损失函数替代Softmax，训练更稳定，零样本性能更优。
- **CoCa**：结合对比学习和生成式目标，同时支持检索和图像描述生成。
- **BLIP-2**：引入Q-Former作为视觉和语言之间的桥梁，参数量更少但性能更强。

在实际工程中，选择嵌入模型时需要考虑以下因素：

1. **输入分辨率**：高分辨率输入能保留更多细节，但计算成本更高。
2. **输出维度**：512维或768维是常见选择，维度越高检索精度越好，但存储成本也更高。
3. **语言支持**：如果处理中文文档，需要选择支持多语言的版本（如Chinese-CLIP）。

```python
from transformers import AutoProcessor, AutoModel
import torch.nn.functional as F

class MultimodalEmbedder:
    """多模态嵌入器，支持图像和文本的统一编码"""
    
    def __init__(self, model_name: str = "BAAI/BGE-VL-MLLM"):
        self.processor = AutoProcessor.from_pretrained(model_name)
        self.model = AutoModel.from_pretrained(model_name)
        self.model.eval()
    
    def embed_images(self, images: list) -> torch.Tensor:
        """批量编码图像"""
        inputs = self.processor(images=images, return_tensors="pt", padding=True)
        with torch.no_grad():
            embeddings = self.model.get_image_features(**inputs)
        return F.normalize(embeddings, p=2, dim=-1)
    
    def embed_texts(self, texts: list) -> torch.Tensor:
        """批量编码文本"""
        inputs = self.processor(text=texts, return_tensors="pt", padding=True, truncation=True)
        with torch.no_grad():
            embeddings = self.model.get_text_features(**inputs)
        return F.normalize(embeddings, p=2, dim=-1)
    
    def cross_modal_retrieval(self, query: str, image_embeddings: torch.Tensor, 
                              image_paths: list, top_k: int = 5) -> list:
        """跨模态检索：文本查图像"""
        query_emb = self.embed_texts([query])
        scores = query_emb @ image_embeddings.T
        top_indices = scores.squeeze(0).topk(top_k).indices.tolist()
        return [image_paths[i] for i in top_indices]

# 构建完整的Multimodal RAG检索流程
def build_multimodal_index(documents: list) -> dict:
    """构建多模态索引"""
    index = {
        "image_embeddings": [],
        "image_paths": [],
        "text_embeddings": [],
        "text_chunks": [],
        "table_embeddings": [],
        "table_chunks": []
    }
    
    embedder = MultimodalEmbedder()
    
    for doc in documents:
        if doc["type"] == "image":
            emb = embedder.embed_images([doc["content"]])
            index["image_embeddings"].append(emb)
            index["image_paths"].append(doc["path"])
        elif doc["type"] == "text":
            emb = embedder.embed_texts([doc["content"]])
            index["text_embeddings"].append(emb)
            index["text_chunks"].append(doc["content"])
        elif doc["type"] == "table":
            # 将表格转换为文本后嵌入
            table_text = table_to_markdown(doc["content"])
            emb = embedder.embed_texts([table_text])
            index["table_embeddings"].append(emb)
            index["table_chunks"].append(table_text)
    
    # 合并所有嵌入到统一空间
    all_embeddings = torch.cat(index["image_embeddings"] + 
                               index["text_embeddings"] + 
                               index["table_embeddings"])
    return {"embeddings": all_embeddings, "index": index}
```

5. **视觉特征提取基础**：了解CNN和ViT的基本原理，知道图像如何被编码为特征向量。

## 实践建议

1. **从CLIP开始快速验证**：不要一开始就追求复杂的多模态方案。使用OpenAI CLIP或开源的SigLIP模型，构建一个简单的图像-文本检索原型，验证你的业务场景是否真正需要多模态能力。这个原型可以在1-2天内完成。

2. **为不同模态设计不同的分块策略**：文本按段落分块，图像按页面或逻辑区域分块，表格保持完整结构。不要试图用统一的分块策略处理所有模态。推荐为每个模态维护独立的索引，在检索时进行多路召回。

3. **优先使用混合检索**：不要只依赖向量相似度。结合基于关键词的稀疏检索（如BM25）和基于语义的稠密检索，可以有效提升召回率。对于表格数据，还可以加入基于规则的精确匹配。

4. **在生成阶段做模态融合**：检索到的多模态信息如何送入大模型是关键。推荐将图像转换为描述文本（使用Caption模型）或直接使用支持多模态输入的大模型（如GPT-4V、Qwen-VL）进行生成。不要试图将图像嵌入直接拼接到文本嵌入中。

5. **建立评估基准**：多模态RAG的评估比纯文本RAG更复杂。除了常规的答案准确性，还需要评估检索到的模态是否匹配查询意图（用户问图，是否返回了图？），以及多模态信息的融合是否自然。建议构建一个小规模的人工标注测试集。
