# 模型A/B测试与灰度发布

## 1. 核心结论
**模型A/B测试与灰度发布是连接模型离线评估与线上商业价值的核心桥梁，其本质是通过科学、可控的线上实验，量化新模型相较于旧模型的真实增量价值，从而驱动模型迭代的理性决策。**

## 2. 为什么重要
在AI工程师的成长路径中，从“能训练出高指标模型”到“能让模型稳定可靠地创造商业价值”是一个关键跃迁。许多团队曾陷入这样的困境：离线评估时AUC、准确率全面领先的新模型，全量上线后关键业务指标（如用户留存、总收入）却不升反降。这是因为离线评估无法完全模拟复杂的线上环境，包括数据分布的变化、用户行为的反馈环路以及与其他系统组件的交互。

A/B测试与灰度发布正是为了解决这一“模型上线黑箱”问题。它们属于**AI工程化与系统**阶段的核心实践，标志着工程师的关注点从单一的算法优化，扩展到包含实验设计、流量管理、指标监控和决策分析的全链路系统工程。掌握这套方法论，意味着你不仅是一个模型构建者，更是一个能够用数据驱动业务增长的价值交付者。

## 3. 核心内容

### 3.1 实验设计：从假设到方案
**概念说明**：实验设计是A/B测试的蓝图，它明确了测试的目标、对象、变量和衡量标准。一个糟糕的实验设计会导致结论无效，甚至产生误导。

**原理解析**：
1.  **明确目标与假设**：首先定义业务目标（如提升点击率），并将其转化为可证伪的统计假设。例如：“新推荐模型（B组）的点击率比旧模型（A组）高。”
2.  **确定核心指标与护栏指标**：
    *   **核心指标**：直接衡量实验目标的指标，如点击率(CTR)、转化率(CVR)。
    *   **护栏指标**：确保实验不会对系统健康度造成负面影响的监控指标，如延迟、错误率、崩溃率。
3.  **确定样本量与实验周期**：使用统计功效分析计算所需的最小样本量，以确保能检测到预期的最小效果提升（MDE）。同时，需考虑周期效应（如周末与工作日的差异），实验应覆盖一个完整的业务周期。

**实践指导**：
设计一个实验文档模板，强制团队在启动测试前思考以下问题：
*   **实验假设**：我们相信 [改变X] 会导致 [指标Y] 发生 [方向Z] 的变化。
*   **实验组与对照组**：对照组（A）使用现有模型，实验组（B）使用新模型。
*   **分流单元**：通常为用户ID或设备ID，确保同一用户在整个实验期内始终处于同一组。
*   **主要指标**：`总点击次数 / 总曝光次数`。
*   **护栏指标**：API P99延迟 < 100ms，错误率 < 0.1%。
*   **样本量估算**：基于当前流量、显著性水平(α=0.05)、统计功效(1-β=0.8)和MDE(0.5%)进行计算。

### 3.2 流量分割：公平与隔离的基石
**概念说明**：流量分割是指将线上用户请求按照一定规则和比例，定向分配到不同实验组（如A/B组）或不同版本（灰度发布）的技术。其核心要求是**均匀性**（样本无偏）和**一致性**（用户粘性）。

**原理解析**：
1.  **分流逻辑**：通常对用户ID或请求ID进行哈希（如MurmurHash），将哈希值取模后映射到不同的“桶”中。例如，将哈希值模100，0-89为A组，90-99为B组（即10%流量）。
2.  **分层与正交**：复杂的系统同时运行多个实验。通过**分层**（不同实验在不同流量层进行）和**正交**（用户在不同层被独立随机分配），可以保证实验之间互不干扰。
3.  **灰度发布**：一种渐进式的流量分割策略。通常分为几个阶段：内部小流量（1%）→ 外部小流量（5%）→ 逐步放量（20%，50%）→ 全量（100%）。每一阶段都进行指标观察，发现问题可快速回滚。

**代码示例**：
以下是一个简化的Python分流服务示例，演示了基于用户ID的哈希分桶逻辑。

```python
import hashlib
import json
from typing import Dict, Any

class TrafficSplitter:
    """一个简单的流量分割器"""
    
    def __init__(self, experiment_config: Dict[str, Any]):
        """
        Args:
            experiment_config: 实验配置，例如：
                {
                    'exp_id': 'rec_model_v2',
                    'buckets': {
                        'A': (0, 8999),  # 90% 流量
                        'B': (9000, 9999) # 10% 流量
                    }
                }
        """
        self.exp_id = experiment_config['exp_id']
        self.buckets = experiment_config['buckets']
        
    def _get_bucket(self, user_id: str) -> str:
        """根据用户ID和实验ID计算所属桶"""
        # 使用实验ID和用户ID共同哈希，确保不同实验分流正交
        hash_input = f"{self.exp_id}:{user_id}".encode('utf-8')
        hash_value = int(hashlib.md5(hash_input).hexdigest(), 16)
        bucket_num = hash_value % 10000  # 分10000个桶
        
        for group, (start, end) in self.buckets.items():
            if start <= bucket_num <= end:
                return group
        # 默认返回对照组，防止异常
        return 'A'
    
    def assign_group(self, user_id: str, request_context: Dict) -> str:
        """分配用户到实验组，可在此处添加更复杂的逻辑（如用户过滤）"""
        # 示例：排除内部测试用户
        if request_context.get('is_internal_user', False):
            return 'A'  # 内部用户始终看到对照组
        
        return self._get_bucket(user_id)

# 使用示例
if __name__ == "__main__":
    config = {
        'exp_id': 'new_rec_model_20231027',
        'buckets': {'A': (0, 8999), 'B': (9000, 9999)}
    }
    splitter = TrafficSplitter(config)
    
    user = "user_12345"
    context = {"is_internal_user": False}
    group = splitter.assign_group(user, context)
    print(f"用户 {user} 被分配到实验组: {group}")
    # 模型服务层根据返回的group决定调用哪个模型
```

### 3.3 指标分析与科学决策
**概念说明**：在实验运行期间及结束后，对收集到的指标数据进行统计分析，以判断新模型（B组）是否显著优于旧模型（A组），并评估其影响幅度。

**原理解析**：
1.  **数据收集与聚合**：确保前端SDK或后端日志能准确上报带有`实验组`标记的曝光、点击等事件。使用数据管道（如Flink, Spark）进行实时/离线聚合，得到各实验组的核心指标值。
2.  **统计显著性检验**：
    *   对于**比例类指标**（如CTR）：通常使用**双比例Z检验**。
    *   对于**均值类指标**（如人均使用时长）：通常使用**T检验**。
    *   检验的目的是计算**p-value**。通常，当p-value < 0.05时，我们拒绝“两组无差异”的原假设，认为差异是统计显著的。
3.  **效应大小与置信区间**：统计显著不代表业务重要。必须关注**效应大小**（如CTR提升了0.8个百分点）及其**置信区间**（如95% CI: [0.5%, 1.1%]）。置信区间提供了效应估计的精度范围。
4.  **多重检验与辛普森悖论**：同时观察多个指标时，可能出现偶然显著的“假阳性”，需进行校正（如Bonferroni校正）。此外，需从不同维度（如新老用户、不同地区）拆解指标，避免因分组不均导致的**辛普森悖论**（整体趋势与子群体趋势相反）。

**实践指导**：
使用Python的`statsmodels`库进行简单的显著性检验分析。

```python
import numpy as np
import pandas as pd
import statsmodels.stats.proportion as proportion

# 模拟A/B两组的曝光点击数据
# 组A: 曝光10000次，点击500次
# 组B: 曝光10000次，点击580次
clicks = np.array([500, 580])
impressions = np.array([10000, 10000])

# 执行双比例Z检验
z_stat, p_value = proportion.proportions_ztest(clicks, impressions)
print(f"Z统计量: {z_stat:.4f}")
print(f"P值: {p_value:.4f}")

if p_value < 0.05:
    print("结果在95%置信水平下统计显著！")
else:
    print("结果不显著。")

# 计算效应大小（CTR提升）及其置信区间
ctr_a = clicks[0] / impressions[0]
ctr_b = clicks[1] / impressions[1]
lift = (ctr_b - ctr_a) / ctr_a
print(f"\n对照组CTR: {ctr_a:.4%}")
print(f("实验组CTR: {ctr_b:.4%}")
print(f"相对提升: {lift:.2%}")

# 使用比例差的置信区间计算
ci_low, ci_high = proportion.confint_proportions_2indep(
    count1=clicks[1], nobs1=impressions[1],
    count2=clicks[0], nobs2=impressions[0],
    method='wald'
)
print(f"CTR绝对差值的95%置信区间: [{ci_low:.4f}, {ci_high:.4f}]")
```

## 4. 知识依赖说明
理解本文内容，需要以下前置知识作为基础：
*   **模型评估（阶段2「机器学习核心」）**：这是A/B测试的“前哨站”。你必须深刻理解离线评估指标（如AUC、F1、LogLoss）的含义与局限，明白为什么离线指标好不代表线上效果好。A/B测试是线上评估的黄金标准，它评估的是模型在真实业务场景下的最终表现。
*   **基础统计学**：包括假设检验、p值、置信区间、统计功效等概念。这是科学分析A/B测试结果，避免做出错误决策的理论根基。
*   **基本的软件工程与系统概念**：了解服务化、API、日志等概念，有助于理解模型如何被部署和集成到线上系统中进行A/B测试。

如果你对上述任何一点感到陌生，建议先回顾相关知识，这将帮助你更顺畅地掌握A/B测试与灰度发布的系统工程全貌。

## 5. 实践建议
1.  **从“实验文化”开始**：推动团队建立“任何主要模型变更必须经过A/B测试”的共识。设计一个简单的实验文档模板，并作为上线流程的强制环节。
2.  **搭建最小可行分流系统**：无需一开始就追求复杂的多层正交系统。可以从一个简单的、基于用户ID哈希的分流服务开始，并确保分流日志被完整记录。这是所有后续分析的基础。
3.  **定义清晰的指标体系**：与产品、业务同学共同确定1-2个**北极星指标**作为核心评估标准，同时定义3-5个关键的护栏指标。避免陷入“观察几十个指标，不知如何决策”的困境。
4.  **重视可视化与自动化报告**：使用Grafana、Tableau等工具搭建实验数据看板，实现核心指标的实时监控。实验结束后，自动生成包含显著性检验结果和效应大小的分析报告。
5.  **建立回滚与降级机制**：在灰度发布方案中，必须预设明确的回滚触发条件（如核心指标下跌X%、错误率飙升）。确保在代码和架构上能实现分钟级的流量切换，将故障影响控制在最小范围。
