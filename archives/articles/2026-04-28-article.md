# Structured Output：让LLM输出结构化数据

**核心结论：** 通过JSON模式、Pydantic验证和约束解码技术，开发者可以强制大语言模型输出符合预定义结构的数据，从而将LLM从“自由文本生成器”转变为“可靠的API后端”，这是构建企业级LLM应用的基石。

## 为什么重要

在AI工程师的成长路径中，**从“让模型说话”到“让模型说对的话”** 是一个关键跃迁。阶段4“大模型应用开发”的核心挑战，正是如何将不可控的文本生成转化为可控的数据管道。想象一下，当你需要从用户对话中提取“姓名、年龄、意向产品”三个字段，或者让LLM生成一个包含10个参数的系统调用指令时，自由文本的模糊性会直接导致下游程序崩溃。

**结构化输出解决了三个根本问题：** 第一，**可靠性**——保证每次输出都能被程序正确解析，无需正则表达式猜谜；第二，**可组合性**——多个LLM调用的输出可以像拼积木一样组合成复杂数据结构；第三，**可验证性**——在进入业务逻辑前，数据格式和类型都能被自动校验。

这项技术将LLM从“聊天机器人”角色升级为“函数调用器”和“数据转换器”。没有它，你构建的智能体系统会像用橡皮泥做的齿轮——看起来能运转，但一受力就变形。掌握结构化输出，意味着你真正理解了如何用工程手段驯服大模型的随机性。

## 核心内容

### JSON模式：最基础的结构化约束

**概念说明：** JSON模式（JSON Schema）是一种声明式语言，用于描述JSON数据的结构和约束。在LLM上下文中，它作为一份“输出说明书”，告诉模型应该生成什么样的JSON对象——包括字段名、类型、是否必填、取值范围等。

**原理解析：** 现代LLM（如GPT-4、Claude 3）在训练过程中已经学习了大量JSON Schema示例。当我们在System Prompt中嵌入JSON Schema时，模型会将其视为**输出格式的硬约束**。关键在于：这不是强制性的（模型仍可能输出非JSON文本），但通过精心设计的Prompt，可以将成功率从50%提升至95%以上。

**代码示例：**
```python
import json
from openai import OpenAI

client = OpenAI(api_key="your-api-key")

# 定义JSON Schema
schema = {
    "type": "object",
    "properties": {
        "name": {"type": "string", "description": "用户姓名"},
        "age": {"type": "integer", "minimum": 0, "maximum": 150},
        "interests": {
            "type": "array",
            "items": {"type": "string"},
            "minItems": 1
        }
    },
    "required": ["name", "age"]
}

response = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "system", "content": f"""
你是一个数据提取助手。请严格按以下JSON Schema输出：
{json.dumps(schema, indent=2)}

只输出JSON对象，不要包含任何其他文字。
"""},
        {"role": "user", "content": "我叫张三，今年25岁，喜欢编程和阅读。"}
    ],
    temperature=0  # 降低随机性
)

# 解析输出
try:
    data = json.loads(response.choices[0].message.content)
    print(f"提取结果: {data}")
except json.JSONDecodeError as e:
    print(f"解析失败: {e}")
```

**实践要点：** 将`temperature`设为0或接近0，这能显著提高格式一致性。同时，在System Prompt中明确“只输出JSON”的指令，并给出Schema示例，效果远好于仅文字描述。

### Pydantic：类型安全的结构化验证

**概念说明：** Pydantic是一个Python库，通过类型注解定义数据模型，并自动进行数据验证和序列化。在LLM应用中，它充当**双重角色**：既是输出格式的定义模板（用于Prompt），又是输出结果的验证器。

**原理解析：** Pydantic的核心优势在于**编译时和运行时的双重保障**。你定义的数据模型（如`UserProfile`）可以：1）自动生成JSON Schema供LLM参考；2）在LLM输出后，自动验证字段类型、范围、必填项。如果输出不符合模型定义，Pydantic会抛出详细的验证错误，告诉你哪个字段出问题以及期望的格式是什么。

**代码示例：**
```python
from pydantic import BaseModel, Field, ValidationError
from typing import List
import json

# 1. 定义Pydantic模型
class UserProfile(BaseModel):
    name: str = Field(..., min_length=1, max_length=50)
    age: int = Field(..., ge=0, le=150)
    interests: List[str] = Field(..., min_length=1)
    email: str | None = Field(None, pattern=r'^[\w\.-]+@[\w\.-]+\.\w+$')

# 2. 自动生成JSON Schema用于Prompt
schema_dict = UserProfile.model_json_schema()
print("生成的Schema:")
print(json.dumps(schema_dict, indent=2))

# 3. 模拟LLM输出（可能是脏数据）
llm_output = '{"name": "张三", "age": 25, "interests": ["编程", "阅读"], "email": "zhangsan@test.com"}'

# 4. 自动验证
try:
    user = UserProfile.model_validate_json(llm_output)
    print(f"验证通过: {user}")
except ValidationError as e:
    print(f"验证失败: {e}")
    # 输出类似：1 validation error for UserProfile
    # age -> 字段'age'的值'abc'不是有效整数
```

**高级用法：** 可以结合`Field`的`description`参数，为每个字段添加自然语言描述，这些描述会自动嵌入生成的JSON Schema中，帮助LLM理解字段含义。

### 约束解码：硬性保证输出格式

**概念说明：** 约束解码（Constrained Decoding）是一种更底层的技术，它**在模型生成每个token时实时干预**，确保生成的序列始终符合预定义语法。与Prompt工程不同，这是从模型输出层强制执行的硬约束。

**原理解析：** 大模型通过“下一个token预测”生成文本。约束解码的原理是：在每一步解码时，**动态计算当前可用的token集合**，只允许生成符合语法规则的token。例如，如果Schema要求下一位置必须是数字，则所有非数字token的概率被强制设为0。这相当于在模型输出层加了一个“语法过滤器”。

**代码示例（使用outlines库）：**
```python
import outlines
from pydantic import BaseModel

# 定义输出结构
class MovieReview(BaseModel):
    title: str
    rating: int  # 1-5
    summary: str

# 创建约束解码模型
model = outlines.models.transformers("gpt2")  # 或使用OpenAI等API

# 生成时自动约束为Pydantic模型格式
generator = outlines.generate.json(model, MovieReview)

# 执行生成
result = generator("请评价电影《流浪地球2》：")
print(f"结构化输出: {result}")
# 输出示例: MovieReview(title='流浪地球2', rating=4, summary='中国科幻电影的里程碑...')

# 验证输出类型
print(f"类型: {type(result)}")  # <class '__main__.MovieReview'>
```

**技术对比：** 约束解码的**优势**是100%格式保证，**劣势**是计算开销略大（每次解码需计算有效token集合），且对某些复杂Schema支持有限。对于生产环境中的关键路径（如金融交易、医疗诊断），推荐使用约束解码；对于一般场景，JSON模式+Pydantic验证的性价比更高。

## 实践建议

1. **从Pydantic开始**：对于新项目，优先使用Pydantic定义数据模型。它提供类型安全、自动验证和Schema生成，是性价比最高的选择。避免手动编写JSON Schema，容易出错且难以维护。

2. **分层使用约束强度**：在开发阶段使用JSON模式+Pydantic验证（灵活调试），在生产环境的关键路径上切换为约束解码（硬性保证）。不要对所有场景都使用约束解码，它会增加延迟和成本。

3. **设计容错机制**：始终假设LLM可能输出无效数据。在代码中加入重试逻辑（重新生成）和降级策略（如使用默认值）。例如：
   ```python
   for attempt in range(3):
       try:
           result = generate_structured_output(prompt)
           validated = MyModel.model_validate(result)
           break
       except (json.JSONDecodeError, ValidationError):
           if attempt == 2:
               validated = MyModel(name="未知", age=0)  # 降级
   ```

4. **利用Field描述**：在Pydantic模型中使用`Field(description=...)`为每个字段添加语义描述。这能显著提升LLM的字段理解准确率，尤其是当字段名不直观时。

5. **记录失败案例**：将LLM输出验证失败的案例记录下来，分析是Prompt问题还是Schema问题。定期用这些案例优化你的Prompt模板，形成“失败-分析-优化”的闭环。
