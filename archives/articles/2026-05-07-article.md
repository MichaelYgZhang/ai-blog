# GraphRAG：图谱增强的检索生成

## 核心结论

**GraphRAG通过将知识图谱的结构化关系引入检索增强生成，在需要多跳推理和全局理解的复杂问答任务中，相比传统RAG（向量检索）实现了显著的准确率提升，是RAG从“语义相似度匹配”迈向“结构化知识推理”的关键进化。**

## 为什么重要

**在RAG进阶的学习路径中，传统RAG（Naive RAG）解决了大模型知识时效性和幻觉的基本问题，但很快暴露了其核心瓶颈：它依赖向量嵌入的语义相似度进行检索，本质上是一种“平面”搜索。** 当用户问“特斯拉的创始人与SpaceX的CEO是什么关系？”时，传统RAG需要分别检索“特斯拉创始人”和“SpaceX CEO”，再将两段文本拼接给LLM。这个过程不仅效率低，而且容易丢失实体间的关联信息，导致答案碎片化甚至错误。

**GraphRAG的出现，正是为了解决这个“关联性”问题。** 它将知识图谱（KG）与RAG结合，利用图结构存储实体和关系，通过图遍历算法（如广度优先搜索、社区检测）进行检索。这使得模型不仅能找到相关文本块，还能理解实体之间的多跳路径和层次结构。**对于AI工程师而言，掌握GraphRAG意味着从“文本检索工程师”升级为“知识推理工程师”，能够构建处理复杂逻辑问答、总结性分析和全局洞察的高级应用。** 在金融、医疗、法律等需要深度关系理解的领域，GraphRAG已成为不可或缺的技术方案。

## 核心内容

### 1. 知识图谱的构建：从非结构化文本到结构化三元组

#### 概念说明
知识图谱（Knowledge Graph）本质上是**实体-关系-实体**的三元组集合，例如（特斯拉，创始人，埃隆·马斯克）。在GraphRAG中，我们需要从原始文档中自动或半自动地抽取出这些三元组，形成图数据库的基础。

#### 原理解析
构建过程通常分为三步：
1. **实体识别（NER）**：识别文本中的命名实体（人名、地名、组织名等）。
2. **关系抽取（RE）**：判断实体对之间是否存在语义关系，并标注关系类型。
3. **实体链接**：将识别出的实体对齐到统一的知识库中（如Wikidata）。

实际应用中，我们常使用大模型本身（如GPT-4）或专门的NLP模型（如BERT+CRF）来完成抽取。为了控制成本，更推荐使用**轻量级模型+规则**的混合方案。

#### 代码示例：使用LLM从文本中抽取三元组

```python
import json
from openai import OpenAI

client = OpenAI(api_key="your-api-key")

def extract_triples(text: str) -> list:
    prompt = f"""
    从以下文本中抽取所有实体-关系-实体的三元组，以JSON列表形式输出。
    格式：[{{"head": "实体1", "relation": "关系", "tail": "实体2"}}]
    文本：{text}
    """
    response = client.chat.completions.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.0
    )
    return json.loads(response.choices[0].message.content)

# 示例文本
doc = "特斯拉的CEO埃隆·马斯克同时也是SpaceX的创始人。"
triples = extract_triples(doc)
print(triples)
# 输出: [{"head": "特斯拉", "relation": "CEO", "tail": "埃隆·马斯克"},
#        {"head": "埃隆·马斯克", "relation": "创始人", "tail": "SpaceX"}]
```

**实践指导**：对于大规模文档，建议使用批处理，并将结果存储到图数据库（如Neo4j）中。注意过滤低置信度的三元组，避免引入噪声。

### 2. 图检索：从向量相似度到图遍历

#### 概念说明
传统RAG的检索基于向量相似度（如余弦相似度），而GraphRAG的检索基于图结构。**核心思想是：将用户问题转化为图查询，通过遍历图中的节点和边来找到最相关的上下文。**

#### 原理解析
图检索有两种主流模式：
- **基于社区的检索**：使用社区检测算法（如Leiden算法）将图划分为多个社区，每个社区代表一个主题簇。对每个社区生成摘要，检索时先匹配社区摘要，再深入社区内部节点。
- **基于路径的检索**：针对需要多跳推理的问题，通过广度优先搜索（BFS）或最短路径算法，找到连接查询实体和目标实体的路径。

**微软的GraphRAG论文**提出了一种混合策略：先利用LLM为每个社区生成全局摘要，检索时同时匹配社区摘要和局部节点，兼顾了全局洞察和局部细节。

#### 代码示例：基于Neo4j的图检索

```python
from neo4j import GraphDatabase

class GraphRetriever:
    def __init__(self, uri, user, password):
        self.driver = GraphDatabase.driver(uri, auth=(user, password))
    
    def retrieve_by_entity(self, entity_name: str, depth: int = 2):
        """
        从指定实体出发，进行深度为depth的图遍历，返回所有关联的节点和关系。
        """
        query = """
        MATCH (n {name: $entity_name})-[r*1..$depth]-(m)
        RETURN n, r, m
        """
        with self.driver.session() as session:
            result = session.run(query, entity_name=entity_name, depth=depth)
            # 处理结果，返回文本化表示
            context = []
            for record in result:
                context.append(f"{record['n']['name']} -[{r['type']}]-> {record['m']['name']}")
            return "\n".join(context)

# 使用示例
retriever = GraphRetriever("bolt://localhost:7687", "neo4j", "password")
context = retriever.retrieve_by_entity("埃隆·马斯克", depth=2)
print(context)
```

**实践指导**：图检索的深度是关键超参数。深度过浅可能遗漏信息，过深则会引入噪声。推荐从`depth=2`开始，根据具体任务调整。同时，可以结合向量检索作为“第一轮召回”，再用图检索进行“第二轮精排”。

### 3. 社区摘要：将局部知识聚合为全局洞察

#### 概念说明
社区摘要（Community Summarization）是GraphRAG区别于传统RAG的核心创新之一。**它通过LLM为每个图社区生成一段自然语言描述，概括该社区的核心主题和实体关系。** 这使得模型能够回答“文档集合的整体主题是什么？”这类全局性问题。

#### 原理解析
实现流程：
1. **社区检测**：使用Leiden算法将图划分为多个社区。
2. **社区摘要生成**：对每个社区，将其包含的所有三元组和关联文本输入LLM，要求生成一段200字以内的摘要。
3. **摘要索引**：将社区摘要向量化，存入向量数据库。
4. **查询时匹配**：用户问题先与社区摘要进行相似度匹配，找到最相关的1-2个社区，再深入这些社区进行局部检索。

#### 代码示例：生成社区摘要

```python
def generate_community_summary(triples: list, llm_client) -> str:
    triples_text = "\n".join([f"{t['head']} - {t['relation']} -> {t['tail']}" for t in triples])
    prompt = f"""
    以下是一个知识图谱社区的实体关系列表。请生成一段简洁的摘要（200字以内），
    描述这个社区的核心主题和关键实体。
    
    关系列表：
    {triples_text}
    
    摘要：
    """
    response = llm_client.chat.completions.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}],
        max_tokens=300
    )
    return response.choices[0].message.content

# 示例：某个社区的局部三元组
community_triples = [
    {"head": "机器学习", "relation": "子领域", "tail": "深度学习"},
    {"head": "深度学习", "relation": "应用", "tail": "图像识别"},
    {"head": "图像识别", "relation": "使用算法", "tail": "CNN"}
]
summary = generate_community_summary(community_triples, client)
print(summary)
# 输出示例："该社区聚焦于机器学习中的深度学习分支，重点描述了深度学习在图像识别领域的应用，核心算法为CNN。"
```

**实践指导**：社区摘要的质量直接决定全局问答效果。建议对生成的摘要进行人工校验，并设置摘要长度上限（200-300字）。对于大型知识图谱，社区数量可能达到数百个，需要设计分层摘要机制（先有高层摘要，再有细节摘要）。

### 4. 混合检索策略：向量 + 图 + 关键词

#### 概念说明
**没有任何一种检索策略是万能的。** 向量检索擅长语义相似度匹配，图检索擅长多跳推理，关键词匹配擅长精确匹配。GraphRAG的最佳实践是**融合多种检索结果**。

#### 原理解析
典型的多路召回融合流程：
1. **向量检索**：将问题向量化，在文档块中检索Top-K。
2. **图检索**：从问题中提取实体，在图数据库中遍历，检索Top-M。
3. **关键词检索**：使用BM25等算法进行精确匹配。
4. **结果融合**：使用RRF（Reciprocal Rank Fusion）或基于LLM的排序器，对多路结果进行重排序。

#### 代码示例：RRF融合

```python
def reciprocal_rank_fusion(results: list, k: int = 60) -> list:
    """
    RRF融合多路检索结果。
    results: 每路检索返回的文档ID列表（按相关性降序排列）。
    """
    scores = {}
    for rank_list in results:
        for rank, doc_id in enumerate(rank_list):
            if doc_id not in scores:
                scores[doc_id] = 0
            scores[doc_id] += 1 / (k + rank + 1)
    # 按分数降序排列
    sorted_docs = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    return [doc_id for doc_id, _ in sorted_docs]

# 示例：三路检索结果
vector_results = ["doc1", "doc3", "doc5"]
graph_results = ["doc2", "doc1", "doc4"]
keyword_results = ["doc1", "doc2", "doc6"]

fused_results = reciprocal_rank_fusion([vector_results, graph_results, keyword_results])
print(fused_results)  # 输出: ['doc1', 'doc2', 'doc3', 'doc4', 'doc5', 'doc6']
```

**实践指导**：RRF简单有效，但无法处理检索结果间的语义冗余。进阶方案是使用**交叉编码器（Cross-Encoder）** 作为重排序器，对候选结果逐对打分，但计算成本较高。实际项目中，建议先用RRF快速上线，再逐步替换为重排序模型。

## 知识依赖说明

**理解本文需要以下前置知识：**

1. **阶段4「RAG进阶」**：熟悉传统RAG的架构（文档分块、向量化、检索、生成），理解向量相似度检索的原理（如余弦相似度、FAISS）。这是理解GraphRAG“为何改进”的基础。
2. **基本图论概念**：了解节点、边、有向图、无向图、路径、社区等基本概念。不需要精通图算法，但需要知道图可以表示实体间的复杂关系。
3. **LLM基础调用**：能够使用OpenAI API或类似接口进行文本生成和结构化输出（如JSON模式）。这是实现三元组抽取和摘要生成的前提。
4. **数据库基础**：了解Neo4j等图数据库的基本操作（创建节点、建立关系、Cypher查询）。不要求成为DBA，但需要能写出简单的图遍历查询。

**建议学习路径**：先完成阶段4的RAG进阶课程，掌握传统RAG的部署和调优；然后学习Neo4j的入门教程（约2小时），最后再回到本文深入GraphRAG实践。

## 实践建议

1. **从小规模开始**：不要一开始就构建包含百万级实体的知识图谱。先用10-20篇文档，手动构建一个小型图谱，验证GraphRAG是否能带来提升。推荐使用`Neo4j Desktop`本地运行。
2. **重视三元组质量**：图检索的效果高度依赖三元组的准确性。建议对LLM抽取的结果进行人工抽检，并建立“置信度阈值”，过滤掉`< 0.7`的三元组。宁可漏掉，不要误判。
3. **先解决“多跳推理”问题**：如果你的应用场景主要是单轮事实问答（如“苹果的股价是多少？”），传统RAG可能已经足够。GraphRAG最适合的场景是“A和B是什么关系？”或“总结文档中所有关于C的讨论”这类需要路径推理的问题。
4. **社区摘要需要迭代**：社区摘要的生成不是一次性的。随着新文档加入，图谱会变化，社区结构也会改变。建议设置**定期重建摘要**的机制（如每周一次），并保留历史版本以便回滚。
5. **监控检索延迟**：图检索的深度和社区数量会显著影响延迟。建议在生产环境中设置超时（如5秒），并启用缓存机制（对常见问题缓存检索结果）。同时，使用异步框架（如FastAPI + asyncio）处理并发请求。

1. **《From Local to Global: A Graph RAG Approach to Query-Focused Summarization》**（微软，2024）：GraphRAG的奠基性论文，详细介绍了社区检测和摘要的算法设计。建议通读全文，特别是实验部分。
2. **Neo4j官方教程《Graph Databases for Beginners》**：快速上手图数据库的最佳资源，包含Cypher查询语言的交互式练习。掌握后可以自己搭建GraphRAG的存储层。
3. **LangChain的GraphRAG集成文档**：LangChain已经提供了GraphRAG的官方实现（`langchain_experimental.graph_rag`），可以直接调用。阅读源码可以理解工程化实现细节，是进阶学习的绝佳材料。
