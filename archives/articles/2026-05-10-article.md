# 大模型应用中的错误处理：构建稳定可靠的AI服务核心

**核心结论：在大模型应用开发中，系统性设计重试策略、降级方案和异常分类机制，是确保AI服务在生产环境中稳定运行、提升用户体验的核心能力。**

## 为什么重要

大模型应用开发与传统软件工程最大的区别在于其**不确定性**。传统API调用通常遵循“输入-输出”的确定性模式，而大模型API却面临着网络波动、模型过载、内容审核拦截、Token限制、响应格式异常等多种非确定性错误。这些错误如果不加处理，轻则导致用户等待超时，重则引发服务雪崩，甚至产生不可控的财务成本。

作为AI工程师，从“能调通API”到“能构建生产级应用”的跃迁，恰恰体现在对错误处理的系统化思考上。**错误处理不是锦上添花，而是大模型应用从原型走向产品的必经之路**。它直接影响服务的可用性SLA、用户体验和运维成本。在阶段4「大模型应用开发」中，掌握错误处理技术意味着你具备了将AI能力工程化的关键能力，能够应对真实场景中的各种边界情况。

## 核心内容

### 异常分类：识别不同类型的错误

在大模型应用中，错误并非“非黑即白”。**正确的异常分类是制定有效处理策略的前提**。根据错误的可恢复性和处理方式，我们可以将其分为三类：

#### 1. 可重试错误（Transient Errors）

这类错误通常是临时性的，稍后重试即可成功。典型场景包括：
- **网络超时**：请求因网络波动未到达服务端
- **服务端限流（Rate Limit）**：API返回429状态码，表明请求频率过高
- **服务端内部错误**：API返回500/503状态码，服务临时不可用

#### 2. 不可重试错误（Non-Transient Errors）

这类错误表明请求本身有问题，重试只会浪费资源。典型场景包括：
- **认证失败**：API Key无效或过期（401错误）
- **参数错误**：模型名称不存在、输入格式不符合要求（400错误）
- **内容违规**：输入或输出触发了内容安全策略（通常返回400或403）

#### 3. 业务逻辑错误（Business Errors）

这类错误与模型输出相关，API调用本身是成功的，但返回的内容不符合业务预期。典型场景包括：
- **响应格式异常**：模型返回了非JSON格式的数据，但业务期望解析JSON
- **内容截断**：输出达到Token限制被截断，导致语义不完整
- **内容质量不达标**：模型输出空字符串、重复内容或明显错误的回答

### 重试策略：从指数退避到抖动

**重试策略的核心思想是：给系统“喘息”的时间**。不加延迟的立即重试往往会导致服务端压力更大，甚至触发更严格的限流。

#### 指数退避（Exponential Backoff）

最经典的重试策略是**指数退避**，即每次重试的等待时间呈指数增长。

```python
import time
import random
from typing import Callable, Any

def retry_with_exponential_backoff(
    func: Callable,
    max_retries: int = 3,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
    retryable_exceptions: tuple = (ConnectionError, TimeoutError)
) -> Any:
    """
    带指数退避的重试装饰器
    
    Args:
        func: 要重试的函数
        max_retries: 最大重试次数
        base_delay: 基础延迟（秒）
        max_delay: 最大延迟（秒）
        retryable_exceptions: 可重试的异常类型
    """
    for attempt in range(max_retries + 1):
        try:
            return func()
        except retryable_exceptions as e:
            if attempt == max_retries:
                raise  # 最后一次重试失败，向上抛出异常
            # 指数退避计算：delay = base_delay * 2^attempt
            delay = min(base_delay * (2 ** attempt), max_delay)
            print(f"重试第 {attempt + 1} 次，等待 {delay:.2f} 秒...")
            time.sleep(delay)
    return None
```

#### 加入抖动（Jitter）

**指数退避的一个问题是“惊群效应”**：当多个请求同时失败时，它们会在相同的时间点重试，导致服务端再次过载。解决方案是加入随机抖动：

```python
def retry_with_jitter(
    func: Callable,
    max_retries: int = 3,
    base_delay: float = 1.0,
    max_delay: float = 60.0
) -> Any:
    """
    带抖动的指数退避重试
    抖动范围：0 到 当前退避时间之间随机
    """
    for attempt in range(max_retries + 1):
        try:
            return func()
        except (ConnectionError, TimeoutError) as e:
            if attempt == max_retries:
                raise
            # 计算退避时间
            backoff = min(base_delay * (2 ** attempt), max_delay)
            # 加入全抖动（Full Jitter）
            delay = random.uniform(0, backoff)
            print(f"重试第 {attempt + 1} 次，等待 {delay:.2f} 秒...")
            time.sleep(delay)
    return None
```

**实践要点**：
- **区分异常类型**：只对可重试错误进行重试，不可重试错误应立即失败
- **设置最大重试次数**：通常3-5次足够，过多重试会导致响应延迟过长
- **结合业务场景**：实时对话场景重试次数应少于后台批量处理场景

### 降级方案：当重试失败时的最后防线

**降级方案的核心理念是：部分功能可用，比完全不可用好**。当大模型服务完全不可用时，我们需要提供备选方案。

#### 1. 缓存降级

对历史请求结果进行缓存，当API不可用时直接返回缓存结果。这是最简单有效的降级策略。

```python
import hashlib
import json
from functools import lru_cache

class LLMServiceWithCache:
    def __init__(self, cache_size: int = 1000):
        self.cache = {}
        self.cache_size = cache_size
    
    def _generate_cache_key(self, prompt: str, model: str, **kwargs) -> str:
        """生成缓存键"""
        content = f"{prompt}:{model}:{json.dumps(kwargs, sort_keys=True)}"
        return hashlib.md5(content.encode()).hexdigest()
    
    def query(self, prompt: str, model: str = "gpt-3.5-turbo", **kwargs) -> str:
        cache_key = self._generate_cache_key(prompt, model, **kwargs)
        
        # 先查缓存
        if cache_key in self.cache:
            print("命中缓存")
            return self.cache[cache_key]
        
        try:
            # 调用大模型API
            response = self._call_llm_api(prompt, model, **kwargs)
            # 写入缓存
            if len(self.cache) >= self.cache_size:
                # 简单淘汰策略：删除最早加入的
                self.cache.pop(next(iter(self.cache)))
            self.cache[cache_key] = response
            return response
        except Exception as e:
            # API失败时，如果有缓存则返回缓存
            if cache_key in self.cache:
                print(f"API调用失败，使用缓存降级: {e}")
                return self.cache[cache_key]
            raise  # 无缓存可用，向上抛出异常
```

#### 2. 模型降级

当主模型（如GPT-4）不可用时，降级到备用模型（如GPT-3.5-turbo或本地模型）。

```python
class ModelDowngradeStrategy:
    def __init__(self):
        # 定义模型优先级列表
        self.model_chain = [
            {"model": "gpt-4", "provider": "openai"},
            {"model": "gpt-3.5-turbo", "provider": "openai"},
            {"model": "claude-instant-1", "provider": "anthropic"},
            {"model": "local-llama-2", "provider": "local"}
        ]
    
    def query_with_downgrade(self, prompt: str) -> str:
        """按优先级依次尝试不同模型"""
        last_error = None
        for model_config in self.model_chain:
            try:
                response = self._call_provider(
                    provider=model_config["provider"],
                    model=model_config["model"],
                    prompt=prompt
                )
                print(f"成功使用模型: {model_config['model']}")
                return response
            except Exception as e:
                last_error = e
                print(f"模型 {model_config['model']} 失败: {e}")
                continue
        
        raise Exception(f"所有模型均不可用，最后错误: {last_error}")
```

#### 3. 简化响应降级

当无法获取完整响应时，返回预设的**固定回复**或**模板回复**。例如，在智能客服场景中，当模型不可用时，返回“您好，我们的AI助手暂时繁忙，请稍后再试，或拨打人工客服电话XXX”。

### 完整错误处理框架

将以上策略整合为一个完整的错误处理框架：

```python
import time
import random
from enum import Enum
from typing import Optional, Callable

class ErrorCategory(Enum):
    RETRYABLE = "retryable"      # 可重试错误
    NON_RETRYABLE = "non_retryable"  # 不可重试错误
    BUSINESS = "business"        # 业务逻辑错误

class LLMErrorHandler:
    def __init__(self, 
                 max_retries: int = 3,
                 fallback_model: Optional[str] = None,
                 cache_enabled: bool = True):
        self.max_retries = max_retries
        self.fallback_model = fallback_model
        self.cache = {} if cache_enabled else None
    
    def classify_error(self, exception: Exception) -> ErrorCategory:
        """分类错误类型"""
        if isinstance(exception, (ConnectionError, TimeoutError)):
            return ErrorCategory.RETRYABLE
        if isinstance(exception, PermissionError):
            return ErrorCategory.NON_RETRYABLE
        # 假设API返回的错误信息中包含特定状态码
        error_msg = str(exception).lower()
        if "rate limit" in error_msg or "429" in error_msg:
            return ErrorCategory.RETRYABLE
        if "invalid" in error_msg or "401" in error_msg or "403" in error_msg:
            return ErrorCategory.NON_RETRYABLE
        return ErrorCategory.BUSINESS
    
    def safe_call(self, func: Callable, *args, **kwargs) -> Any:
        """安全的调用函数，集成重试、降级和缓存"""
        # 尝试从缓存获取
        if self.cache is not None:
            cache_key = str((args, kwargs))
            if cache_key in self.cache:
                return self.cache[cache_key]
        
        for attempt in range(self.max_retries + 1):
            try:
                result = func(*args, **kwargs)
                # 缓存结果
                if self.cache is not None:
                    self.cache[cache_key] = result
                return result
            except Exception as e:
                category = self.classify_error(e)
                
                if category == ErrorCategory.NON_RETRYABLE:
                    raise  # 不可重试，直接抛出
                
                if attempt == self.max_retries:
                    # 最后一次重试失败，尝试降级
                    if self.fallback_model:
                        print(f"主模型失败，降级到备用模型: {self.fallback_model}")
                        return self._call_fallback(func, *args, **kwargs)
                    raise
                
                # 可重试错误，等待后重试
                delay = random.uniform(0, min(1.0 * (2 ** attempt), 60.0))
                print(f"第 {attempt + 1} 次重试，等待 {delay:.2f} 秒...")
                time.sleep(delay)
        
        return None
```

## 实践建议

1. **从日志开始**：在开发阶段就为每个API调用添加结构化日志，记录请求参数、响应时间、错误类型。这是后续优化错误处理策略的数据基础。

2. **为每个API调用设置超时**：大模型API响应时间波动很大，**务必设置合理的超时时间**（通常30-60秒），避免请求长时间挂起。

3. **区分开发环境和生产环境**：开发环境可以关闭重试机制以快速暴露错误，生产环境则需要完整的重试和降级策略。

4. **监控重试次数**：如果某个API调用频繁触发重试，说明存在系统性问题（如网络不稳定、限流策略过严），需要从根源解决而非仅靠重试。

5. **为降级策略设置熔断机制**：当连续失败达到阈值（如10次/分钟），自动切换到降级模式，避免持续调用已失效的服务。
