# Text-to-SQL：自然语言查询数据库

## 核心结论

**Text-to-SQL技术使非技术用户能够通过自然语言直接查询数据库，其核心挑战在于将模糊的人类语言精准映射为结构化的SQL查询，而大模型的出现使这一过程的准确率从传统方法的60%提升至85%以上。**

## 为什么重要

在AI工程师的成长路径中，Text-to-SQL是一个极具代表性的**大模型应用范式**。它融合了自然语言理解、结构化知识表达和精确推理三大核心能力，是检验大模型“理解+生成”双重能力的试金石。当企业积累了大量关系型数据，却只有少数数据工程师能熟练使用SQL时，Text-to-SQL就成为了打通“数据民主化”最后一公里的关键工具。

对于AI工程师而言，掌握Text-to-SQL不仅是学会调用大模型API，更意味着理解如何将**结构化知识（数据库Schema）**与**非结构化指令（自然语言问题）**进行有效对齐。这项技术直接解决了企业中普遍存在的“数据可用但不可及”的痛点：业务人员需要查询“上个月华东区销售额最高的产品是什么”，而数据团队疲于应对大量类似的临时查询需求。Text-to-SQL让业务人员直接与数据对话，大幅降低数据获取成本。

## 核心内容

### Schema理解：让模型读懂数据库结构

#### 概念说明

Schema理解是Text-to-SQL的第一步，指让模型理解数据库中有哪些表、每张表包含哪些字段、字段的数据类型以及表之间的关联关系。这是**将数据库的结构化元数据转化为模型可理解的上下文**的过程。

#### 原理解析

大模型无法直接“看”数据库，我们需要将Schema信息转化为文本描述。关键在于**有选择性地提供相关信息**——一个大型数据库可能包含上百张表，全部输入会导致上下文过长且引入噪声。实践中采用“Schema筛选+结构化描述”的策略：

1. **Schema筛选**：基于用户问题中的关键词（如“销售额”“华东区”），通过向量检索或规则匹配，定位到可能相关的表和字段
2. **结构化描述**：将筛选后的Schema转化为模型熟悉的自然语言格式

#### 代码示例

```python
from typing import List, Dict

def build_schema_prompt(tables: List[Dict], question: str) -> str:
    """
    将数据库Schema转化为模型可理解的提示词
    """
    # 1. Schema筛选：基于问题关键词匹配相关表
    keywords = extract_keywords(question)  # 简化：假设已有分词工具
    relevant_tables = []
    for table in tables:
        table_text = table['name'] + ' ' + ' '.join([col['name'] for col in table['columns']])
        if any(kw in table_text for kw in keywords):
            relevant_tables.append(table)
    
    # 2. 构建结构化描述
    schema_desc = "数据库包含以下相关表：\n"
    for table in relevant_tables:
        schema_desc += f"表名: {table['name']}\n"
        schema_desc += "字段:\n"
        for col in table['columns']:
            schema_desc += f"  - {col['name']} ({col['type']}): {col.get('description', '')}\n"
        
        # 处理外键关系
        if 'foreign_keys' in table:
            schema_desc += "外键关系:\n"
            for fk in table['foreign_keys']:
                schema_desc += f"  - {fk['column']} -> {fk['ref_table']}.{fk['ref_column']}\n"
    
    return schema_desc

# 示例使用
database_schema = [
    {
        'name': 'orders',
        'columns': [
            {'name': 'order_id', 'type': 'INT', 'description': '订单ID'},
            {'name': 'amount', 'type': 'FLOAT', 'description': '订单金额'},
            {'name': 'region', 'type': 'VARCHAR', 'description': '销售区域'},
            {'name': 'order_date', 'type': 'DATE', 'description': '订单日期'}
        ],
        'foreign_keys': []
    },
    {
        'name': 'products',
        'columns': [
            {'name': 'product_id', 'type': 'INT'},
            {'name': 'product_name', 'type': 'VARCHAR', 'description': '产品名称'}
        ],
        'foreign_keys': [
            {'column': 'product_id', 'ref_table': 'orders', 'ref_column': 'product_id'}
        ]
    }
]

question = "上个月华东区销售额最高的产品是什么？"
schema_prompt = build_schema_prompt(database_schema, question)
print(schema_prompt)
# 输出：数据库包含以下相关表：
# 表名: orders
# 字段:
#   - order_id (INT): 订单ID
#   - amount (FLOAT): 订单金额
#   - region (VARCHAR): 销售区域
#   - order_date (DATE): 订单日期
# 表名: products
# ...
```

### SQL生成：从自然语言到结构化查询

#### 概念说明

SQL生成是核心环节，**将用户问题与Schema上下文共同输入大模型，生成可执行的SQL语句**。这一过程需要模型同时理解自然语言的语义和SQL的语法约束。

#### 原理解析

高质量SQL生成依赖三个关键设计：

1. **Few-shot示例**：提供2-3个同数据库的问答示例，引导模型理解查询模式
2. **格式约束**：明确要求输出格式（如“只输出SQL语句，不要解释”），避免模型输出冗余内容
3. **分步推理**：对于复杂查询（包含子查询、多表JOIN），引导模型先拆解问题再生成SQL

实践中发现，**直接生成最终SQL往往不如先生成“执行计划”再转换为SQL准确**。例如，对于“查询每个部门工资最高的员工”，模型先输出“先按部门分组，再在每个组内按工资排序取第一条”，再生成对应的SQL。

#### 代码示例

```python
import openai

def generate_sql(question: str, schema_prompt: str) -> str:
    """
    基于问题和Schema生成SQL
    """
    # Few-shot示例
    few_shot_examples = """
示例1:
问题：查询所有订单的总金额
SQL: SELECT SUM(amount) FROM orders;

示例2:
问题：查询2023年的订单数量
SQL: SELECT COUNT(*) FROM orders WHERE YEAR(order_date) = 2023;
"""
    
    prompt = f"""你是一个SQL专家。请根据数据库Schema和用户问题，生成正确的SQL查询。

{schema_prompt}

{few_shot_examples}

用户问题：{question}

请只输出SQL语句，不要包含任何解释。如果需要多步操作，请先输出思考过程（用--注释标注），再输出最终SQL。

思考过程：
-- 1. 首先需要理解问题：查询上个月（2024年1月）华东区的销售数据
-- 2. 需要关联orders和products表，通过product_id
-- 3. 按产品分组计算销售额，取最高值
最终SQL：
"""
    
    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[
            {"role": "system", "content": "你是一个SQL专家，擅长编写精确的SQL查询。"},
            {"role": "user", "content": prompt}
        ],
        temperature=0.1  # 低温度确保确定性输出
    )
    
    sql = response.choices[0].message.content
    # 提取最终SQL（去除注释部分）
    if "最终SQL：" in sql:
        sql = sql.split("最终SQL：")[-1].strip()
    
    return sql

# 示例调用
sql = generate_sql(
    "上个月华东区销售额最高的产品是什么？",
    schema_prompt
)
print(sql)
# 输出：SELECT p.product_name, SUM(o.amount) as total_amount
# FROM orders o
# JOIN products p ON o.product_id = p.product_id
# WHERE o.region = '华东区' AND o.order_date BETWEEN '2024-01-01' AND '2024-01-31'
# GROUP BY p.product_name
# ORDER BY total_amount DESC
# LIMIT 1;
```

### 结果验证：确保SQL正确执行

#### 概念说明

结果验证是Text-to-SQL系统的**安全网**。由于大模型可能生成语法错误或语义错误的SQL，我们需要在执行前进行校验，并在执行后验证结果合理性。

#### 原理解析

验证分为三个层次：

1. **语法验证**：使用SQL解析器检查语法正确性，避免执行失败
2. **语义验证**：检查SQL是否“合理”——例如SELECT子句中的字段是否在FROM子句的表中存在，JOIN条件是否正确
3. **结果验证**：执行SQL后，检验结果是否与问题语义一致。例如，如果问“数量”，结果中不应出现金额

**关键技巧**：对于涉及聚合或排序的查询，可以生成一个“简化版”SQL（去掉LIMIT、GROUP BY等）来验证基础数据的正确性。

#### 代码示例

```python
import sqlparse
from sql_metadata import Parser

def validate_sql(sql: str, schema: Dict) -> Dict:
    """
    验证SQL的正确性和安全性
    返回验证结果字典
    """
    result = {
        'is_valid': True,
        'errors': [],
        'warnings': []
    }
    
    # 1. 语法检查
    try:
        parsed = sqlparse.parse(sql)
        if not parsed:
            result['is_valid'] = False
            result['errors'].append("SQL解析失败：空语句")
            return result
    except Exception as e:
        result['is_valid'] = False
        result['errors'].append(f"SQL语法错误：{str(e)}")
        return result
    
    # 2. 语义检查：验证表和字段存在
    try:
        parser = Parser(sql)
        tables = parser.tables
        columns = parser.columns
        
        # 检查表是否存在
        for table in tables:
            if table not in [t['name'] for t in schema]:
                result['warnings'].append(f"表 '{table}' 在Schema中未找到")
        
        # 检查字段是否存在（简化实现）
        # 实际应用中需要更精确的字段-表映射
    except Exception as e:
        result['warnings'].append(f"语义分析异常：{str(e)}")
    
    # 3. 安全检查：防止危险操作
    dangerous_keywords = ['DROP', 'DELETE', 'UPDATE', 'INSERT', 'ALTER']
    for keyword in dangerous_keywords:
        if keyword in sql.upper():
            result['is_valid'] = False
            result['errors'].append(f"包含危险操作：{keyword}")
            break
    
    return result

def verify_result(question: str, sql: str, result: List) -> bool:
    """
    验证执行结果是否与问题语义一致
    """
    # 简单规则：检查结果字段名是否与问题关键词匹配
    # 实际应用中可以再次调用LLM进行语义验证
    question_lower = question.lower()
    result_fields = list(result[0].keys()) if result else []
    
    # 如果问题提到“数量”，结果中应该有count相关的字段
    if '数量' in question_lower or '多少' in question_lower:
        if not any('count' in field.lower() for field in result_fields):
            return False
    
    # 如果问题提到“最高”或“最大”，结果应该按降序排列
    if '最高' in question_lower or '最大' in question_lower:
        # 检查SQL中是否有ORDER BY ... DESC
        if 'ORDER BY' in sql.upper() and 'DESC' not in sql.upper():
            return False
    
    return True

# 完整执行流程
def text_to_sql_pipeline(question: str, db_schema: Dict, db_connection) -> str:
    """
    完整的Text-to-SQL执行流水线
    """
    # Step 1: Schema理解
    schema_prompt = build_schema_prompt(db_schema, question)
    
    # Step 2: SQL生成
    sql = generate_sql(question, schema_prompt)
    print(f"生成的SQL: {sql}")
    
    # Step 3: 验证
    validation = validate_sql(sql, db_schema)
    if not validation['is_valid']:
        return f"SQL验证失败: {validation['errors']}"
    
    # Step 4: 执行
    try:
        cursor = db_connection.cursor()
        cursor.execute(sql)
        result = cursor.fetchall()
        
        # Step 5: 结果验证
        if verify_result(question, sql, result):
            return result
        else:
            return "生成的SQL执行成功，但结果与问题语义可能不一致，请检查"
            
    except Exception as e:
        return f"SQL执行错误: {str(e)}"
```

## 知识依赖说明

理解本文需要以下前置知识：

1. **SQL基础**：熟悉SELECT、JOIN、GROUP BY、ORDER BY等常见SQL语法，这是评估生成结果正确性的基础
2. **大模型API调用**：了解如何调用OpenAI、Claude等大模型API，特别是提示工程（Prompt Engineering）的基本技巧
3. **Python编程**：能够阅读和理解Python代码，特别是数据库连接（如sqlite3、psycopg2）的基本操作

建议先学习阶段4「大模型应用开发」中的「大模型应用」课程，掌握大模型的基本调用和提示设计方法，再深入学习Text-to-SQL的专项技术。

## 实践建议

1. **从简单查询开始**：先让模型处理单表查询（如“查询所有订单”），再逐步增加复杂度（多表JOIN、子查询），建立信心后再投入生产环境

2. **建立Schema描述规范**：为每个字段添加清晰的中文描述，特别是枚举值（如region字段的取值：华东、华北、华南），这能显著提升生成准确率

3. **实现人机协同的验证流程**：对于高价值查询，让生成的SQL先经过人工审核再执行。可以设计一个“草稿-审核-执行”的三步流程

4. **构建领域特定的Few-shot库**：收集业务中常见的20-30个查询问题及其正确SQL，作为Few-shot示例的种子库。定期更新以覆盖新的查询模式

5. **监控并记录失败案例**：当生成的SQL执行失败或结果错误时，记录完整上下文（问题、Schema、生成的SQL、错误信息），用于后续优化提示模板或微调模型

## 延伸阅读

1. **Spider数据集**：耶鲁大学发布的Text-to-SQL基准数据集，包含200个数据库和超过10,000个自然语言问题，是评估和对比不同方法的标准平台

2. **LangChain SQL Agent**：LangChain框架中的SQL代理实现，展示了如何结合大模型和数据库操作的最佳实践，包括自动错误重试和查询优化

3. **DAIL-SQL论文**：2023年发表的《DAIL-SQL: Learning to Generate SQL from Natural Language with Domain-Aware In-Context Learning》，详细介绍了如何通过领域感知的上下文学习提升Text-to-SQL性能
